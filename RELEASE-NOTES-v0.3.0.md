# Release Notes — v0.3.0

## Summary

Kid Chat MVP v0.3.0 adds the first focused task-and-reward learning loop.

This release does not attempt to generalize into a full task engine. Instead, it ships a narrow, practical MVP for one concrete scenario: a short French-writing task generated by 大智, delivered to George or Grace, evaluated inside chat, and rewarded with a theme-matched image when completed.

## Highlights

### French-writing task MVP

- a parent can trigger a French-writing task directly from the child chat header
- the current task prompt is shown inline in the chat experience
- the first MVP task shape is intentionally simple:
  - write a short French text
  - around 20 characters/words-worth of length guidance
  - on a clear theme such as rockets

### In-chat evaluation loop

- when a French-writing task is active, the child can submit their writing directly in chat
- the app checks a few lightweight completion conditions before rewarding:
  - does it look like French?
  - does it stay on topic?
  - is the length roughly near the target?
- if the submission is close but not complete, the child gets a gentle retry message instead of a hard failure

### Reward image delivery

- when the task is completed, Kid Chat generates a polished reward image matched to the task theme
- the reward image is delivered back inside the same chat thread
- this keeps assignment, submission, evaluation, and reward in one simple flow

## Scope note

This is a deliberately narrow MVP release.

v0.3.0 is meant to validate the interaction model before building broader systems such as:

- generic task types
- scoring tiers
- reward ladders
- certificates and badges
- multi-step evaluation rubrics

## Recommended follow-up after release

- observe whether children naturally understand the assignment → submit → reward loop
- refine the French-detection and topic-detection heuristics with real family usage
- decide whether the next task themes should stay fixed or become configurable per child
- consider whether reward images should be stored locally instead of depending on provider-returned URLs for every path
